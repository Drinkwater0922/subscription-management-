# PennyLoop v1.0.1 Requirements — Feedback Entry + Privacy-Safe Analytics

Status: planned for the first post-review update. Do not modify the App Store build that is currently in review unless Apple rejects it or a critical launch blocker appears.

## Goal

Add two narrow capabilities for v1.0.1:

1. A visible feedback/support entry inside the app.
2. A privacy-safe analytics foundation that records product funnel events without collecting subscription content, screenshots, personal identifiers, or advertising identifiers.

This should help diagnose real user friction after launch while preserving PennyLoop's current privacy promise: local-first, no account, no ads, and no analytics SDK in the first shipping analytics pass.

## Non-Goals

- Do not add Firebase, PostHog, Amplitude, Segment, AppsFlyer, Adjust, or any other third-party analytics SDK in this pass.
- Do not add login, account creation, user profile, or server-side identity.
- Do not upload subscription names, prices, notes, URLs, OCR text, screenshots, or category details.
- Do not change the current one-time Lifetime IAP model.
- Do not change App Store metadata for the build currently in review.

## Requirement 1 — Feedback Entry

### User Story

As a user, I can open Settings and find a clear way to send feedback or request support, especially for purchase restore, screenshot import, reminders, and iCloud sync issues.

### Product Behavior

- Add a feedback row to `SettingsView`.
- Location: near the existing legal links section, below Terms of Service.
- Label:
  - English: `FEEDBACK`
  - Simplified Chinese: `意见反馈`
- Tapping it should open the published support page:
  - `https://drinkwater0922.github.io/subscription-management-/support/`
- The support page already links users to a GitHub issue creation page. The app should not embed a web view for v1.0.1; use the system browser via SwiftUI `Link` or `openURL`.

### Implementation Notes

Suggested changes:

- `Trackr/Core/Brand/BrandConfig.swift`
  - Add `supportURL = URL(string: "https://drinkwater0922.github.io/subscription-management-/support/")!`
- `Trackr/Features/Settings/SettingsView.swift`
  - Add the feedback link in `linksSection`.
  - Keep visual style consistent with Privacy Policy / Terms of Service rows.
- `Trackr/Resources/Localizable.xcstrings`
  - Add localization key for `FEEDBACK` with `zh-Hans` value `意见反馈`.

### Acceptance Criteria

- Settings shows Privacy Policy, Terms of Service, and Feedback.
- Tapping Feedback opens the support URL.
- The Settings snapshot updates cleanly.
- Existing Settings behavior for privacy, terms, restore purchases, and Pro status still works.

## Requirement 2 — Privacy-Safe Analytics Foundation

### Strategy

Add an internal analytics abstraction and wire key funnel events to it. For v1.0.1, the default production implementation should be a no-op or local console logger only. This lets us review event shape and code boundaries before deciding whether to enable remote analytics in a later version.

Recommended default:

- Debug builds: log events to console.
- Release builds: no-op.

Do not send events over the network in v1.0.1 unless the privacy policy and App Privacy answers are explicitly updated before submission.

### Event Whitelist

Only these event names are allowed in v1.0.1:

| Event | Trigger |
|---|---|
| `app_launched` | App root appears for the first time in a process |
| `onboarding_completed` | User completes onboarding |
| `settings_opened` | Settings sheet appears |
| `feedback_tapped` | User taps Feedback in Settings |
| `subscription_add_started` | Add subscription sheet opens |
| `subscription_added` | A subscription is successfully created |
| `photo_import_started` | User starts Import from Photo |
| `photo_import_succeeded` | Photo import produces at least one usable draft |
| `photo_import_failed` | Photo import fails or produces no usable draft |
| `paywall_shown` | Paywall is presented |
| `purchase_started` | User starts a StoreKit purchase |
| `purchase_succeeded` | StoreKit returns a successful Pro entitlement |
| `purchase_failed` | StoreKit purchase fails or is cancelled |
| `restore_started` | User taps Restore Purchases |
| `restore_succeeded` | Entitlement refresh resolves to Pro |

If an event is not listed here, do not add it in v1.0.1.

### Allowed Properties

Only these event properties are allowed:

- `app_version`
- `build`
- `locale`
- `pro_status` (`free`, `proLifetime`)
- `source` (`manual`, `preset`, `photo`, `settings`, `subscription_limit`, `insights_locked`, `manual_upgrade`)
- `result` (`success`, `failed`, `cancelled`, `empty`)
- `error_code` from a controlled enum, such as `validation_failed`, `ocr_empty`, `ocr_error`, `storekit_cancelled`, `storekit_failed`

### Forbidden Properties

Never include:

- Subscription name
- Subscription price or amount
- Currency for a specific subscription
- Billing date or renewal date
- Category
- Notes
- URL
- OCR text
- Screenshot image data
- User email
- Apple ID
- Device advertising ID
- Persistent custom user ID
- IP address collection by app code

### Suggested Code Shape

Add a small analytics module:

```text
Trackr/Core/Analytics/
  AnalyticsClient.swift
  AnalyticsEvent.swift
  AnalyticsEnvironment.swift
  SystemAnalyticsClient.swift
```

Recommended API shape:

```swift
protocol AnalyticsClient {
    func track(_ event: AnalyticsEvent)
}

struct AnalyticsEvent: Equatable {
    let name: AnalyticsEventName
    let properties: [AnalyticsPropertyKey: String]
}

enum AnalyticsEventName: String {
    case appLaunched = "app_launched"
    case onboardingCompleted = "onboarding_completed"
    case settingsOpened = "settings_opened"
    case feedbackTapped = "feedback_tapped"
    case subscriptionAddStarted = "subscription_add_started"
    case subscriptionAdded = "subscription_added"
    case photoImportStarted = "photo_import_started"
    case photoImportSucceeded = "photo_import_succeeded"
    case photoImportFailed = "photo_import_failed"
    case paywallShown = "paywall_shown"
    case purchaseStarted = "purchase_started"
    case purchaseSucceeded = "purchase_succeeded"
    case purchaseFailed = "purchase_failed"
    case restoreStarted = "restore_started"
    case restoreSucceeded = "restore_succeeded"
}
```

Use a SwiftUI environment key so views do not instantiate analytics directly:

```swift
@Environment(\.analytics) private var analytics
```

Provide a `RecordingAnalyticsClient` in tests.

### Initial Wiring Sites

Wire only low-risk events first:

- `TrackrApp.swift`
  - Inject analytics client into environment.
  - Track `app_launched` once from the root surface.
- `RootView.completeOnboarding()`
  - Track `onboarding_completed` after saving completion timestamp.
- `SettingsView`
  - Track `settings_opened` on appear.
  - Track `feedback_tapped` when the feedback link is tapped.
  - Track `restore_started` when Restore Purchases is tapped.
  - Track `restore_succeeded` after refresh if entitlement is Pro.
- Add subscription flow
  - Track `subscription_added` only after successful insert.
  - Use `source` only from controlled values: `manual`, `preset`, `photo`.
- Photo import flow
  - Track start/success/failure without sending any OCR text or image-derived content.
- Paywall / purchase flow
  - Track paywall shown and purchase result without product price or storefront.

If a wiring site is hard to reach cleanly, leave it unwired and document the gap in the PR description rather than forcing a broad refactor.

## Privacy and App Store Impact

For v1.0.1 with console/no-op analytics:

- Privacy Policy does not need to claim third-party analytics.
- App Privacy does not need analytics data collection.
- App description can keep the "no analytics SDK" wording.

If a remote analytics backend is introduced later:

- Update `docs/legal/privacy.md`.
- Update App Store Connect App Privacy answers.
- Update `docs/release/app-store-listing-en.md`.
- Update `docs/release/app-store-listing-zh-Hans.md`.
- Re-review forbidden properties before shipping.

## Review Checklist

The implementation is acceptable when:

- Feedback entry is visible and opens a useful support path.
- Analytics is behind an injectable client.
- Release behavior does not send remote analytics in v1.0.1.
- Event names and properties are constrained by enums/types.
- No subscription content or screenshot-derived content can enter analytics.
- Tests cover BrandConfig support URL, Settings rendering, and analytics event shape.
- Current App Store review build is not disturbed unless explicitly requested.
