# PennyLoop v1.2 Requirements — Insights Redesign

Status: planned for v1.2. Ships after v1.1 (Home + Detail closed-loop + FX rework + PriceHistoryEntry) is live on the App Store. Do not bundle into v1.1 — that scope is locked.

Reference prototype: `docs/design/prototype-insights.html` (commit `d660093` on the `v1.1` branch). The PRD below is the authoritative spec; the prototype is the visual reference.

## Goal

Replace the current Insights view (three totals cards: MONTHLY / YEARLY / ACTIVE SUBS) with an action-oriented "what to cancel" dashboard. The new Insights answers the keep/kill question across the whole subscription list — the same mental model the v1.1 Detail page applies to one subscription, scaled to all of them.

Five modules: a NEXT 30 DAYS DUE hero, a demoted stat strip, a top-5 suspect ranking with context tags, a category breakdown, and a per-currency breakdown. All five behind the existing Pro gate.

## Thesis Change

| | Old Insights | New Insights |
|---|---|---|
| Question answered | "How much do I spend?" | "What should I cancel?" |
| Primary content | Three totals | Top-5 ranked, most-suspect subscriptions |
| Tone | Diagnostic / passive | Action-oriented |
| Pro upgrade pitch | "See your totals" | "See which 5 subscriptions to cancel first" |

The new thesis is consistent with the v1.1 Home + Detail keep/kill loop and with the product's North Star ("a subscription companion that lowers anxiety instead of adding to it") — surfacing what to drop is more anxiety-reducing than reporting totals.

## Non-Goals

- Do **not** integrate with iOS Screen Time / DeviceActivity APIs (kept for ROADMAP Phase 2 once there are real users and signal on what to build).
- Do **not** add bundle / consolidate detection (e.g. "you have Netflix + Disney+ + HBO").
- Do **not** add CSV / PDF export.
- Do **not** add year-over-year trends (no historical snapshots yet — Tier 2 work, requires monthly snapshotting infrastructure that lands separately).
- Do **not** add a year-in-review shareable card (deferred to v1.3 or v2.0).
- Do **not** add a "Mark as used" manual usage signal on the subscription (deferred — would need a new field on `Subscription`).
- Do **not** change the Pro entitlement gate — Insights stays Pro-only as a single unit.
- Do **not** change app pricing or the one-time Lifetime IAP model.
- Do **not** add per-subscription cost-per-day display on Home or Detail rows. That's a different surface and could land separately; this PRD only covers Insights.

## Dependencies

- **v1.1 must be live on the App Store first.** This PRD assumes the v1.1 FX rework has landed: `MonthlyTotalCalculator` reads `FXRateTable` and converts at display time; `FXRateTableRepository.convert(amount:from:to:)` is available and `nonisolated`.
- `PriceHistoryEntry` (v1.1 B1) is not strictly required but is in place. The `stagnation` signal in the suspect score uses `Subscription.updatedAt`, not the price history.
- `Subscription.trialEndsAt` and `isTrial(at:)` (added between v1.1 sessions) are used by Requirement 3 to handle trials in the ranking.

## Requirement 1 — Hero: NEXT 30 DAYS DUE

### User Story

As a Pro user, I open Insights and immediately see how much money is about to leave my account in the next 30 days, so I can react before the charge lands instead of seeing it on a statement after the fact.

### Product Behavior

- Sum `amount` of active subscriptions whose `nextBillingDate` falls in `[now, now + 30 days]`.
- Exclude paused subscriptions (`isActive == false` or `pausedUntil` in the future).
- Trials (`isTrial(at: now) == true`) **are** included if their conversion charge falls within the window. Use their `amount` as the conversion-charge value.
- Convert each contribution from the subscription's `currency` to the user's display currency via `FXRateTableRepository.convert(...)`. If conversion returns `nil` for a sub (rare — its currency isn't in the cached table), skip that sub and add it to a hidden diagnostic count (do not surface to the user in v1.2).
- Subtitle: "N charges incoming" where N = the count of subscriptions contributing to the total.
- The hero replaces the current MONTHLY card as the primary heading.

### Acceptance Criteria

- Display matches the prototype's hero block.
- Pausing a subscription removes it from the total within one render cycle.
- Changing the display currency re-renders the amount through the cache table.
- A subscription whose `nextBillingDate` is 30 days from now (inclusive) **is** included.
- A subscription whose `nextBillingDate` is 31 days from now is **not** included.

## Requirement 2 — Currency Switcher

### User Story

As a Pro user with multi-currency subscriptions, I can switch the displayed currency on the Insights page without going into Settings, so I can quickly view my totals in whichever currency is most meaningful for me right now.

### Product Behavior

- Three pill buttons at the top of the Insights view: `USD $`, `CNY ¥`, `EUR €`.
- The currently-selected pill is highlighted (`TrackrColors.accent` border + foreground).
- Tapping a pill **persists** the choice to `UserSettings.defaultCurrency` — the user's choice carries over to Home, Detail, and future opens of Insights. Switching here is equivalent to changing the default currency in Settings.
- All amounts on the page re-render via the cached FX table when the choice changes.
- If a chosen currency is missing from the cached `FXRateTable`, fall back to the user's previous selection and surface no error (the user might see this happen if they just installed and the bundle fallback doesn't include a region they want; v1.2 doesn't address this beyond graceful no-op).

### Acceptance Criteria

- Switching pills updates the hero, stat strip, ranking row amounts, category bars, and currency-row "≈/yr" approximations in one render.
- Re-opening Insights shows the last-selected currency.
- Going to Settings → Default Currency reflects the last-selected currency.

## Requirement 3 — Suspect Ranking (Top 5)

### User Story

As a Pro user, I see five subscriptions ranked most-likely-to-cancel-first so I know where to look first instead of scanning my whole list.

### Product Behavior

#### Scoring formula

For each candidate subscription:

```
score = priceWeight + imminenceWeight + stagnationWeight    // total 0–100
priceWeight       = (monthlyContribution_in_displayCurrency / max_monthly) * 40
imminenceWeight   = max(0, 1 - daysUntilNextBilling / 30) * 30
stagnationWeight  = min(daysSinceUpdatedAt / 180, 1) * 30
```

Where:

- `monthlyContribution_in_displayCurrency` is the same value `MonthlyTotalCalculator` uses for the home total (cycle-normalized + currency-converted).
- `max_monthly` is the maximum `monthlyContribution_in_displayCurrency` across the candidate set; if `max_monthly == 0`, all `priceWeight`s are 0.
- `daysUntilNextBilling` is `Calendar.current.dateComponents([.day], from: now, to: sub.nextBillingDate).day ?? 0`.
- `daysSinceUpdatedAt` is the same calculation against `sub.updatedAt`.

#### Candidate set

Include a subscription when **all** of:

- `isActive == true` and (`pausedUntil == nil` or `pausedUntil < now`).
- `nextBillingDate >= now` (don't rank subs whose next billing is already overdue / not refreshed — they'll be caught by Home's RENEWS IN tag, not Insights).
- If `isTrial(at: now) == true`, include using the post-trial `amount` (the imminent charge value).

#### Output

- Sort by `score` descending. Tie-break by `monthlyContribution_in_displayCurrency` descending; second tie-break by `name` ascending for stability.
- Take the top 5. If fewer than 5 candidates exist, show only what's available.
- If 0 candidates exist (rare — paused-everything edge case), show an empty-state message: `「No active subscriptions to rank.」` / `「没有活跃订阅可排名。」`

#### Row display

Each row contains:

1. Rank number (1–5) in `TrackrColors.accent`.
2. Subscription name in the standard list-row style.
3. Cost line: `<amount>/mo · <amount>/天` (or `/day` in English). The `/mo` value is the `monthlyContribution` in display currency. The `/day` value is `/mo ÷ 30`.
4. Up to **two** tags, drawn in order of priority. Show a tag when its corresponding weight exceeds the threshold below:

| Tag | Threshold | Style | Text (en / zh) |
|---|---|---|---|
| `EXPENSIVE` | `priceWeight > 25` | accent border, accent text | `EXPENSIVE` / `贵` |
| `RENEWS IN X DAYS` | `imminenceWeight > 15` | warn border, warn text | `RENEWS IN X DAYS` / `X 天内续费` |
| `NOT TOUCHED IN X DAYS` | `stagnationWeight > 15` | muted border + text | `NOT TOUCHED IN X DAYS` / `X 天没动过` |

If 3 tags qualify, show the 2 with the highest individual weight contribution.

#### Interaction

- Tapping a row navigates to that subscription's Detail page (the v1.1 Detail redesign). Standard navigation; no special transition.

### Acceptance Criteria

- Given a fixed test data set, the ranker returns the same top-5 in the same order across runs (deterministic).
- A paused subscription does not appear in the ranking.
- A subscription whose `nextBillingDate` is in the past does not appear.
- Trial subscriptions appear if their `trialEndsAt` is in the future and their post-trial `amount > 0`.
- A subscription with `score > 0` but no qualifying tags shows zero tags (do not invent labels).
- Tapping any row reaches Detail without an intermediate sheet.

## Requirement 4 — Category Breakdown

### User Story

As a Pro user, I see how my monthly spend splits across categories so I can spot concentration (e.g. "I spend 45% on streaming") without doing the math myself.

### Product Behavior

- Group active subscriptions by `Category` (the existing enum).
- For each category, sum `monthlyContribution_in_displayCurrency`.
- Sort categories by sum descending.
- Render one row per category as a horizontal fill bar:
  - Top line: category name (left) + amount + percentage of total (right).
  - Below: a 6-pixel-tall fill bar. Track in `TrackrColors.bg3`. Fill in `TrackrColors.accent`. Width = `(category_sum / total) * 100%`.
- Threshold: render this section only when **≥ 2 categories** are represented in the active set. Single-category users (everyone in `.streaming`, etc.) get no breakdown.

### Acceptance Criteria

- Percentages sum to 100 (with standard rounding).
- Switching display currency re-renders amounts but leaves percentages unchanged.
- Hiding/pausing a subscription updates its category total (and removes the category row entirely if it was the only one in that category).

## Requirement 5 — Per-Currency Breakdown

### User Story

As a Pro user with subscriptions billed in multiple currencies, I see my monthly spend split by original billing currency so I understand my real currency exposure (e.g. "$128/mo in USD subscriptions, ¥18/mo in CNY").

### Product Behavior

- Group active subscriptions by their **original** `currency` (the field stored on the subscription, not the display currency).
- For each currency, sum the cycle-normalized monthly contribution **in that currency** (no FX conversion for the main number).
- Right column shows an "≈ /yr" approximation **in the display currency**, computed via `FXRateTableRepository.convert(...)` × 12. Truncate to no decimals.
- Sort by monthly contribution descending, where comparison is done via the display-currency equivalent (so the row with the larger real value comes first).
- Render: `[currency code]   [original amount /mo]   [≈ display-currency /yr]`.
- Threshold: render this section only when **≥ 2 distinct currencies** are represented. Single-currency users (everyone in CNY, etc.) get no breakdown.

### Acceptance Criteria

- The main per-row amount stays in the subscription's original currency regardless of display-currency switching.
- The "≈ /yr" approximation re-renders when display currency switches.
- A subscription whose original currency is missing from the FX cache contributes to its own currency row normally (no conversion needed for the main number); the "≈ /yr" cell for that currency shows `—` (em dash) instead of an amount.

## 目标

把现在的 Insights(三张卡:MONTHLY / YEARLY / ACTIVE SUBS)替换成一个「该砍什么」导向的行动型 dashboard。新 Insights 把 v1.1 详情页的 keep/kill 决策模型从「单条订阅」放大到「全部订阅」,正面回答用户最该问的那个问题:**「这堆订阅里,我先看哪几个?」**

| 主内容 | 三个总额数字 | 最该看一眼的 5 个订阅 |
| 调性 | 诊断 / 被动 | 行动导向 |

## Requirement 1 —— Hero:未来 30 天扣款

### 用户故事

作为 Pro 用户,我打开 Insights 就立刻看到「未来 30 天会被扣多少钱」,这样我能**在扣款前**反应过来,而不是事后看账单才发现。

### 产品行为

- 计算所有活跃订阅里,`nextBillingDate` 落在 `[now, now + 30 天]` 区间的 `amount` 求和
- 排除暂停的订阅(`isActive == false` 或 `pausedUntil` 在未来)
- 试用订阅(`isTrial(at: now) == true`)**计入**,如果它的转正扣款落在这 30 天窗口内,用它转正后的 `amount` 作为扣款额
- 每条订阅的贡献都通过 `FXRateTableRepository.convert(...)` 从原币种换算到显示币种。如果某条转换返回 `nil`(罕见 —— 它的币种不在缓存表里),跳过这条订阅,加入隐藏的诊断计数(v1.2 不对用户暴露)
- 副标题:`N charges incoming`,N = 计入总额的订阅数
- 这个 hero 替换掉现在的 MONTHLY 卡作为页面主标题

### 验收标准

- 显示和原型 hero 区块一致
- 暂停一条订阅后,一次渲染周期内从总额里移除
- 切换显示币种,金额通过缓存表重新换算
- `nextBillingDate` 距今正好 30 天的订阅 **计入**(含端点)
- `nextBillingDate` 距今 31 天的订阅 **不计入**

## Requirement 2 —— 币种切换器

### 用户故事

作为有多币种订阅的 Pro 用户,我在 Insights 页面上直接切显示币种,不用钻进设置,**快速用对我此刻最有意义的那个币种看总额**。

### 产品行为

- Insights 顶部三个 pill 按钮:`USD $`、`CNY ¥`、`EUR €`
- 当前选中那个 pill 高亮(`TrackrColors.accent` 边 + 字)
- 点击 pill **持久化**到 `UserSettings.defaultCurrency` —— 用户选的币种会带到首页、详情、之后再打开 Insights 都生效。**等价于在设置里改默认币种**
- 切换后,页面上所有金额通过缓存 FX 表重新渲染
- 如果选的币种不在缓存 `FXRateTable` 里,回退到上一次的选择,**不报错也不弹提示**(用户可能刚装、bundled fallback 不含他想要的区域;v1.2 不处理这个边角,优雅 no-op)

### 验收标准

- 切 pill 后,hero / stat 条 / 排行榜金额 / 分类条 / 币种行的「≈/yr」近似值都在一次渲染内更新
- 重新打开 Insights,显示的是上次选的币种
- 进设置 → Default Currency,看到的是上次选的币种

## Requirement 3 —— Suspect 排行榜(Top 5)

### 用户故事

作为 Pro 用户,我看到 5 个**最该优先砍掉**的订阅,这样我知道**先看哪几个**,不用扫整个列表。

### 产品行为

#### 计分公式

对每条候选订阅:

```
score = priceWeight + imminenceWeight + stagnationWeight    // 总分 0–100
priceWeight       = (显示币种下的月成本 / 最大月成本) * 40
imminenceWeight   = max(0, 1 - 距下次扣款天数 / 30) * 30
stagnationWeight  = min(距 updatedAt 天数 / 180, 1) * 30
```

其中:

- 「显示币种下的月成本」用的就是 `MonthlyTotalCalculator` 给主页总额用的那个值(周期归一化 + 币种换算)
- 「最大月成本」是候选集里所有订阅月成本的最大值;如果最大值为 0,所有 `priceWeight` 都为 0
- 「距下次扣款天数」= `Calendar.current.dateComponents([.day], from: now, to: sub.nextBillingDate).day ?? 0`
- 「距 updatedAt 天数」同理,对照 `sub.updatedAt`

#### 候选集

订阅进候选,需要**同时满足**:

- `isActive == true`,且(`pausedUntil == nil` 或 `pausedUntil < now`)
- `nextBillingDate >= now`(不排已过期/未刷新的订阅 —— 首页的 RENEWS IN 标签会处理它们,Insights 不重复)
- 如果 `isTrial(at: now) == true`,用转正后的 `amount` 计入(就是即将发生的扣款金额)

#### 输出

- 按 `score` 降序排序。同分时,以「显示币种下的月成本」降序作为 tie-break;再同分以 `name` 升序保证稳定
- 取前 5。候选不足 5 时,有多少显示多少
- 候选为 0(罕见 —— 全部暂停的边角),显示空态文案:`「No active subscriptions to rank.」` / `「没有活跃订阅可排名。」`

#### 行渲染

每行包含:

1. 排名数字(1–5),用 `TrackrColors.accent`
2. 订阅名,标准列表行样式
3. 成本行:`<金额>/mo · <金额>/天`。`/mo` = 显示币种下的月成本;`/天` = `/mo ÷ 30`
4. 最多 **2 个**标签,按优先级排。某个 weight 超过下面阈值时显示对应标签:

| 标签 | 阈值 | 样式 | 文案(en / zh) |
|---|---|---|---|
| `EXPENSIVE` | `priceWeight > 25` | 亮绿边、亮绿字 | `EXPENSIVE` / `贵` |
| `RENEWS IN X DAYS` | `imminenceWeight > 15` | 警告红边、警告红字 | `RENEWS IN X DAYS` / `X 天内续费` |
| `NOT TOUCHED IN X DAYS` | `stagnationWeight > 15` | 灰边、灰字 | `NOT TOUCHED IN X DAYS` / `X 天没动过` |

3 个标签都达标时,取**单项 weight 贡献最大**的 2 个显示。

#### 交互

- 点击一行 → 跳到那条订阅的详情页(v1.1 详情页重做版)。标准导航,**无特殊转场**

### 验收标准

- 给定固定测试数据集,排行榜 top-5 顺序在多次运行间一致(确定性)
- 暂停的订阅**不**出现在排行榜
- `nextBillingDate` 在过去的订阅**不**出现
- 试用订阅在 `trialEndsAt` 在未来 且 转正 `amount > 0` 时出现
- 一条 `score > 0` 但没任何 tag 达标的订阅,显示 0 个 tag(**不要硬造标签**)
- 点击任意一行能到达详情页,**中间没有 sheet**

## Requirement 4 —— 按分类拆解

### 用户故事

作为 Pro 用户,我看到月开销在各分类间的分布,这样我能**一眼看出集中**(比如「我 45% 都花在流媒体上」),不用自己算。

### 产品行为

- 按 `Category` 枚举(已有)对活跃订阅分组
- 每个分类求「显示币种下的月成本」总和
- 按总额降序排
- 每个分类一行,水平 fill bar:
  - 顶行:分类名(左)+ 金额 + 占总额百分比(右)
  - 下面:6 像素高的 fill bar。track 用 `TrackrColors.bg3`,fill 用 `TrackrColors.accent`。宽度 = `(该分类总额 / 总计) * 100%`
- 门槛:**≥ 2 个分类**才渲染这个版块。单分类用户(全都是 `.streaming`)不显示

### 验收标准

- 百分比加起来 = 100(标准四舍五入)
- 切显示币种,金额变,**百分比不变**
- 隐藏/暂停某条订阅,该分类的总额随之更新(如果是该分类最后一条,该行整体消失)

## Requirement 5 —— 按币种拆解

### 用户故事

作为有多币种订阅的 Pro 用户,我看到月开销按**原始计费币种**分组,**理解我真实的币种暴露**(比如「USD 订阅 $128/mo,CNY 订阅 ¥18/mo」)。

### 产品行为

- 按订阅的**原始** `currency`(订阅上存的字段,**不是**显示币种)分组
- 每个币种求周期归一化的月成本,**以该币种本身为单位**(不做 FX 换算)
- 右列展示「≈ /yr」近似,**以显示币种为单位**,通过 `FXRateTableRepository.convert(...)` × 12 算出。**整数,无小数**
- 按月成本降序排,比较时用「换算到显示币种」的等价值(这样真实金额大的排前面)
- 渲染:`[币种代码]   [原币种 /mo]   [≈ 显示币种 /yr]`
- 门槛:**≥ 2 种 currency**才渲染这个版块。单币种用户(全 CNY)不显示

### 验收标准

- 主中列金额永远是订阅的原币种,**不随显示币种切换变**
- 「≈ /yr」近似列在切显示币种时重新渲染
- 某条订阅的原币种不在 FX 缓存里 —— 该币种行的主中列正常显示(原币种,不需要换算),「≈ /yr」格子显示 `—`(em dash)
