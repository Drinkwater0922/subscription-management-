# PennyLoop v1.3 Requirements — Cancelled Subscriptions

Status: planned for v1.3. Ships after v1.2 (Insights redesign) is live on the App Store. Do not bundle into v1.2 — that scope is locked and on the TestFlight track.

This PRD overturns Open Question 1 from `docs/design/2026-05-21-home-detail-closed-loop.md`. That decision ("keep only PAUSE and DELETE; do not add a CANCELLED state") was made on theoretical grounds ("pause already covers it"). Real-world usage by the author (2026-05-26: deleted a just-ended subscription, immediately felt the data loss as an emotional gap) disproved it. v1.3 corrects the model.

Reference competitor: Orbit ships exactly this UX — a "Mark as Cancelled / Subscribed" toggle plus a separate destructive "Delete" action. v1.3 adopts the same two-action model and **removes PAUSE entirely**.

## Goal

Add a persisted CANCELLED state to subscriptions so users can stop counting an ended subscription against current spend **without losing its history**. Enables future retrospection ("Netflix cost me ¥1,044 over 18 months"), the year-in-review shareable card on the ROADMAP, and the "I cancelled X subs this year, saving ¥Y" identity-reinforcement story.

Simplifies the state machine from three actions (PAUSE / DELETE) plus a temporal pause modifier down to two actions (toggle CANCELLED ↔ ACTIVE / DELETE) — matching the proven Orbit pattern.

## Thesis Change

| | Today (v1.2) | v1.3 |
|---|---|---|
| State machine | ACTIVE ↔ PAUSED (with optional auto-resume date) + DELETE | ACTIVE ↔ CANCELLED + DELETE |
| Detail action area | PAUSE button + DELETE button | "Mark as Cancelled / Mark as Subscribed" toggle + DELETE (with confirm) |
| "I cancelled a sub" outcome | Either pause indefinitely (semantically wrong: pause = will come back) or delete (loses history) | Mark as Cancelled — kept in history, out of current totals, one-tap reversible |
| Year-in-review feasibility | Blocked by data loss when users delete ended subs | Unblocked — cancelled subs retain price + duration + category |

The product is now honest about three distinct user intents (active / closed-but-remembered / never-existed) instead of conflating them into two.

## Non-Goals

- Do **not** add a separate "Archived" / "Suspended" / "Inactive" state on top of CANCELLED. Two states is the entire point; three is what we're moving away from.
- Do **not** add bulk-cancel UI ("cancel 5 subscriptions at once"). Not needed pre-launch; revisit if support requests it.
- Do **not** add cancellation-reason capture ("why did you cancel?"). Adds friction, low signal, deferred.
- Do **not** add the year-in-review shareable card itself. v1.3 unblocks it by preserving data; the card is a separate feature for v1.4 or v2.0.
- Do **not** ship Screen Time / DeviceActivity integration. Still Phase 2 ROADMAP.
- Do **not** change app pricing or the one-time Lifetime IAP model.
- Do **not** Pro-gate the CANCELLED feature itself. Marking as cancelled is core hygiene; free users get it.

## Dependencies

- **v1.2 must be live on the App Store first.** v1.3 work happens on a `v1.3` branch cut off `v1.2` after v1.2 ships.
- No new external dependencies. No new entitlements, no new permissions.

## State Machine

```
                  ┌──── mark as cancelled ────┐
                  │                            ▼
    [ ACTIVE ]                          [ CANCELLED ]
                  ▲                            │
                  └──── mark as subscribed ────┘
                  
          ──── delete ──── (destructive, both states; confirmation required)
                  │
                  ▼
          (row removed from store, no recovery)
```

- Status is a derived computed property over `cancelledAt: Date?`:
  - `cancelledAt == nil` → ACTIVE
  - `cancelledAt != nil` → CANCELLED
- No PAUSED state. The `pausedUntil` field is migrated to `cancelledAt` on first launch (see Migration & Compatibility).

## Layout Changes

### Home

Add a new `CANCELLED` group, mirroring the visual treatment of `FREE TRIALS` and `ACTIVE`:

```
┌─ Hero (existing) ──────────────────────────┐
├─ FREE TRIALS (existing) ────────────────────┤
├─ ACTIVE (existing) ─────────────────────────┤
├─ CANCELLED ─────────────────────────────────┤
│  (collapsed by default; tap to expand)     │
│  Header shows count + lifetime spend       │
│  Rows: dimmed, "Cancelled DD MMM" replaces │
│  the "Renews in N days" line               │
└────────────────────────────────────────────┘
```

The CANCELLED group is **hidden entirely when count == 0** to keep first-time users' Home clean.

### Detail

Action area changes from two buttons to two buttons with new copy:

```
[ MARK AS CANCELLED ]   ← active subs (lime / accent)
[ DELETE ]              ← red / warn, with confirmation

[ MARK AS SUBSCRIBED ]  ← cancelled subs (lime / accent)
[ DELETE ]              ← red / warn, with confirmation
```

When CANCELLED:
- Reading body shows `CANCELLED ON 24 MAY 2026` in muted color where the countdown would be.
- Price history list stays as-is (history is the whole point of preserving the sub).
- Tapping the URL still works (lets users go un-cancel at the service).

### Insights

- CANCELLED subs are excluded from: NEXT 30 DAYS DUE, suspect ranking, THIS MONTH, THIS YEAR, ACTIVE count, BY CATEGORY, BY CURRENCY.
- Add one new stat to the strip: **`LIFETIME` / `历来累计`** — total amount paid across every subscription ever (active + cancelled), summed in display currency. This is the "you've actually spent this much on subscriptions" memory.

## Requirement 1 — Model: Add `cancelledAt`, remove `pausedUntil` (and isActive semantics simplification)

### User Story

As the engineering team, we need a single source of truth for the active/cancelled distinction so every consumer (Home, Detail, Insights, notifications, widgets) reads from the same field.

### Product Behavior

- Add `cancelledAt: Date?` to `Subscription`. Nullable; `nil` means active.
- Keep `isActive: Bool` for now to avoid breaking widgets/notifications that read it; rewrite it as a computed property derived from `cancelledAt == nil`. (Pure additive at the schema level — no destructive SwiftData migration.)
- `pausedUntil: Date?` is migrated to `cancelledAt` on first v1.3 launch (see Migration), then becomes dead code. Remove its read sites in v1.3; keep the stored property on the model so SwiftData doesn't reject loading old data. Delete the stored property in v1.4 or v1.5 after the field has been unused for a release.

### Acceptance Criteria

- `Subscription.cancelledAt` round-trips through SwiftData persistence.
- New `Subscription.isCancelled` computed property returns `cancelledAt != nil`.
- All call sites of `pausedUntil` in v1.2 (verified via grep) are removed or rewritten to consult `cancelledAt`.
- Cascade-delete on `priceHistory` still works after a sub is cancelled then deleted.

## Requirement 2 — Detail page: toggle + destructive delete with confirmation

### User Story

As a user, when I'm done with a subscription, I tap one clear action that preserves my history. If I tapped wrong, I can flip it back with one more tap. Only when I really mean it do I lose the data.

### Product Behavior

- Detail action area renders two buttons (lime / warn) per the layout above. Button labels switch on state:
  - Active: `MARK AS CANCELLED` (lime) + `DELETE` (warn)
  - Cancelled: `MARK AS SUBSCRIBED` (lime) + `DELETE` (warn)
- `MARK AS CANCELLED` sets `cancelledAt = .now`, calls `NotificationCoordinator.refresh()` (cancelling any pending renewal notification for this sub), and updates `updatedAt`.
- `MARK AS SUBSCRIBED` sets `cancelledAt = nil`, recomputes `nextBillingDate` if it has fallen into the past (advance to the next valid renewal at the original cycle), and re-schedules renewal notifications via the coordinator.
- `DELETE` now requires a one-step confirmation dialog: `"Delete this subscription? Its history and price changes will be lost. This cannot be undone."` Confirming runs the existing delete path.
- The Detail reading view header replaces the countdown with `CANCELLED ON DD MMM YYYY` in `TrackrColors.fg3` when the sub is cancelled.
- The Detail edit form stays functional for cancelled subs (user might want to correct a name or amount in their history).

### Acceptance Criteria

- Toggling cancelled → active → cancelled in rapid succession leaves no orphan notifications.
- Deleting a sub still requires confirmation; tapping outside the dialog cancels the deletion.
- A cancelled sub whose `nextBillingDate` is in the past, then un-cancelled, has its `nextBillingDate` advanced to the next valid cycle date (not silently overdue).
- Snapshot tests cover both states' reading body.

## Requirement 3 — Home: CANCELLED group

### User Story

As a user, I see my cancelled subscriptions at the bottom of Home so I can browse my "alumni" without them cluttering my current view.

### Product Behavior

- Add a new group below `ACTIVE`, header: `CANCELLED` + count + lifetime-spend total (in display currency).
- Default state: **collapsed**. Tap header to expand.
- Hidden entirely when count == 0.
- Rows in the group render at reduced opacity (`TrackrColors.fg2` for name, `TrackrColors.fg3` for amount) and show `"CANCELLED <DD MMM>"` in place of the renewal countdown.
- Tapping a row opens Detail (which renders the cancelled-state version of itself).

### Acceptance Criteria

- New section appears immediately after the first sub is cancelled, disappears immediately after the last cancelled sub is un-cancelled or deleted.
- Expand / collapse state is page-local (resets on next app open) — keeping the cancelled section folded by default each session is intentional.
- Tapping a cancelled row navigates to Detail without delay.
- Snapshot baselines for Home (populated + empty cancelled group).

## Requirement 4 — Insights: filter cancelled subs + LIFETIME stat

### User Story

As a Pro user, my Insights numbers reflect what I'm currently paying — cancelled subs don't inflate my current spend — but I can also see how much I've **ever** paid on subscriptions, as a measure of the financial weight I've decided to walk away from.

### Product Behavior

- Every existing v1.2 Insights calculator must skip subs where `isCancelled == true`:
  - `MonthlyTotalCalculator` (verify already-correct for `isActive == false` once `isActive` becomes derived)
  - `UpcomingChargesCalculator`
  - `SuspectRanker`
  - `CategoryBreakdown`
  - `CurrencyBreakdown`
- Add new pure calculator `LifetimeSpendCalculator.total(of:in:rateTable:)` that sums **all** subscriptions (active + cancelled), each contributing `monthlyEquivalent × monthsBetween(startDate, cancelledAt ?? now)`. Returns a `Decimal` in the supplied display currency.
- Add a fourth column to the stat strip: `LIFETIME` / `历来累计`. Renders the calculator's output in display currency.
- This stat reads from active + cancelled subs; deleted subs are still gone (no resurrection from the void).

### Acceptance Criteria

- Cancelling a subscription removes it from MONTHLY / YEARLY / NEXT 30 DAYS / suspect / category / currency within one render cycle.
- The same cancellation increments `LIFETIME` by `monthlyEquivalent × (months between startDate and cancelledAt)`.
- Un-cancelling reverses the above.
- `LifetimeSpendCalculator` has unit tests covering: active-only, cancelled-only, mixed, foreign-currency conversion, missing-rate skip, empty input.

---

## Requirement 5 — Migration: existing PAUSED state → CANCELLED

### User Story

As an existing TestFlight / App Store user with paused subscriptions in my data, my paused subs become "cancelled" on first v1.3 launch — preserving their data and surfacing them in the new CANCELLED group, instead of vanishing or staying in a now-defunct state.

### Product Behavior

A one-time backfill runs at app launch (modeled on the `.initial` price-history backfill already shipped in v1.1). For each subscription:

| Current state | After migration |
|---|---|
| `isActive == true && pausedUntil == nil` | `cancelledAt = nil` (active) |
| `isActive == true && pausedUntil > now` | `cancelledAt = pausedUntil` (cancelled, anchor the date at the user's intended pause end) |
| `isActive == true && pausedUntil < now` | `cancelledAt = nil` (active; pause already expired) |
| `isActive == false` | `cancelledAt = updatedAt` (cancelled, anchor at the most recent edit) |

The migration runs once, idempotent (guarded by a flag in `UserSettings`, e.g. `v1_3_pausedBackfillDoneAt: Date?`).

### Acceptance Criteria

- A pre-v1.3 store with one of each scenario migrates correctly.
- Running the migration twice produces the same result as running it once.
- Migration is bounded in time (synchronous, blocking the launch path only as long as needed — historically <100ms for hundreds of subs).
- Tests construct a SwiftData container in each pre-migration state and assert the post-migration field values.

## Pro Gating

- `MARK AS CANCELLED / SUBSCRIBED` and the CANCELLED Home group are **not Pro-gated** — they're core data-hygiene features every user needs.
- The new `LIFETIME` stat in Insights is **Pro-gated** alongside the rest of the Insights view (no separate gate).
- Pricing model unchanged.

## Migration & Compatibility

- Schema additions are additive (`cancelledAt: Date?`). No destructive SwiftData migration.
- TestFlight users' data is migrated by the backfill above. Lossless.
- Widget extension: continue reading `isActive` (which is now derived). No widget code changes needed in v1.3.
- Existing Localizable strings: add `INSIGHTS_LIFETIME_STAT` / `HOME_CANCELLED_GROUP_HEADER` / detail-action button copy / delete-confirmation copy via `InsightsCopy` (or a new `CancellationCopy` peer).

## Open Questions (must resolve before eng)

1. **Cancellation button copy.** Proposed: `MARK AS CANCELLED` ↔ `MARK AS SUBSCRIBED` (mirrors Orbit's "标记为已取消 / 已订阅"). Alternatives: `CANCEL` ↔ `RESUBSCRIBE`, `CLOSE` ↔ `REOPEN`. Recommend the proposed version — "Mark as" frames it as a label change, lowering the perceived irreversibility.
2. **Delete confirmation copy.** Proposed: `"Delete this subscription? Its history and price changes will be lost. This cannot be undone."` Confirm or rewrite.
3. **CANCELLED group default state — collapsed or expanded.** Recommend **collapsed** — keeps Home clean; users who want to browse alumni do one tap.
4. **LIFETIME stat in Insights — Pro-gated or free?** Recommend **Pro-gated** alongside the rest of Insights — keeps the Pro upgrade story coherent ("Pro = see your patterns, not just your present"). Alternative: free, as a teaser.
5. **Notifications on cancellation.** Recommend cancelling all pending renewal notifications for the sub when it's marked cancelled. Re-creating them on un-cancellation. Any edge case (e.g. notification already fired this morning) is fine to let through.
6. **Migration anchor for `isActive == false` subs.** Proposed: `cancelledAt = updatedAt`. Alternative: `cancelledAt = .now` (records the migration moment rather than the original deactivation). Recommend `updatedAt` for historical truthfulness.

## Acceptance Criteria (rollup)

- Two-action Detail UI per the spec; toggle is reversible and visually obvious.
- Cancelled subs preserved in store; visible in the Home CANCELLED group; excluded from every "current money" view.
- New `LIFETIME` Insights stat reflects active + cancelled total.
- Migration of all four legacy state combinations covered by tests.
- All existing tests pass (currently 452 on `v1.2`).
- New unit tests: `LifetimeSpendCalculator_Tests`, `Subscription_Cancellation_Tests`, `PausedToCancelledMigration_Tests`.
- Updated snapshot tests: `HomeView_Snapshot_Tests` (cancelled group populated + empty), `SubscriptionDetailView_Snapshot_Tests` (both states), `InsightsView_Snapshot_Tests` (re-record for new stat column).

## Out of Scope for v1.3 — Do Not Sneak In

- Bulk cancel / multi-select.
- Cancellation reason capture.
- Year-in-review shareable card (v1.4 candidate, built on this data).
- Auto-cancel detection (e.g. "we noticed you haven't tapped X in 6 months — auto-cancel?").
- Notification copy enhancements ("Cancelled on … reactivate now?").
- Server-driven preset catalog (separate PRD).
- Any change to Pro pricing or the upgrade flow.

# PennyLoop v1.3 需求文档 —— 已取消订阅

状态:计划在 v1.3 落地。**必须在 v1.2(Insights 重做)上线 App Store 之后再开工。** 不要塞进 v1.2,scope 已锁、且正在 TestFlight 流程上。

本 PRD **推翻** `docs/design/2026-05-21-home-detail-closed-loop.md` 里的待解问题 1(「只保留 PAUSE 和 DELETE,不新增 CANCELLED 态」)。那个决定是凭推理做的(「PAUSE 已经覆盖了」)。作者在 2026-05-26 的真实使用证明它覆盖不了 —— 删掉一条刚到期的订阅后,**立刻感受到数据丢失的情感空白**。v1.3 修正这个模型。

参考竞品:**Orbit 正好做了这个 UX —— 一个「标记为已取消 / 已订阅」的切换 + 一个独立的「删除订阅」**。v1.3 采纳同款两动作模型,**完全移除 PAUSE**。

## 目标

给订阅加一个**持久化的 CANCELLED 状态**,让用户能停止把一条已结束的订阅算进当前花费,**但保留它的历史**。这为以后做以下功能解锁了基础数据:

- 回顾(「Netflix 这 18 个月一共扣了我 ¥1,044」)
- 年度分享卡(ROADMAP 已有)
- 「我今年砍了 X 个订阅,省了 ¥Y」这种身份强化叙事

同时把状态机从「三动作(PAUSE / DELETE)+ 时间型暂停」**简化为「两动作(CANCELLED ↔ ACTIVE 切换 / DELETE)」**,匹配 Orbit 验证过的模式。

## thesis 变化

| | v1.2 现状 | v1.3 |
|---|---|---|
| 状态机 | ACTIVE ↔ PAUSED(可选自动恢复日期)+ DELETE | ACTIVE ↔ CANCELLED + DELETE |
| 详情页操作区 | PAUSE 按钮 + DELETE 按钮 | 「标记为已取消 / 已订阅」切换 + DELETE(带确认) |
| 「我取消了一个订阅」结果 | 要么无限期 pause(语义错位:pause = 会回来),要么 delete(丢历史) | 标记为已取消 —— 留在历史里、不计入当前总额、一键可逆 |
| 年度回顾可行性 | 被「用户砍掉时删数据」堵死 | 解锁 —— 已取消订阅的价格 / 时长 / 分类全留 |

产品现在对三种不同用户意图(在用 / 关闭但记得 / 从未存在)**诚实区分**,不再把它们硬塞进两个桶。

## 非目标(明确不做)

- **不**加第三个状态(「Archived」/「Suspended」/「Inactive」)叠在 CANCELLED 之上。两态是核心 —— 三态正是我们要逃离的
- **不**做批量取消 UI(「一次取消 5 条」)。上线前用不上,用户提需求再说
- **不**做取消原因捕获(「你为什么取消?」)。增加摩擦、信号弱,推后
- **不**在 v1.3 做年度回顾分享卡。v1.3 通过保留数据为它**解锁**;卡片本身是 v1.4 或 v2.0 的独立功能
- **不**接入 Screen Time / DeviceActivity。仍然是 ROADMAP Phase 2
- **不**改 app 定价或一次性 Lifetime IAP 模式
- **不** Pro-gate CANCELLED 功能本身。标记取消是核心数据卫生,免费用户也得有

## 依赖

- **v1.2 必须先上 App Store。** v1.3 工作在从 v1.2 切出来的 `v1.3` 分支上进行
- 无新外部依赖。无新权限、无新 entitlement

## 状态机

```
                  ┌──── 标记为已取消 ────┐
                  │                      ▼
    [ ACTIVE ]                    [ CANCELLED ]
                  ▲                      │
                  └──── 标记为已订阅 ────┘
                  
          ──── 删除 ──── (两态都可,需确认对话框)
                  │
                  ▼
          (记录从存储里抹掉,无法恢复)
```

- 状态是一个 derived computed property,基于 `cancelledAt: Date?`:
  - `cancelledAt == nil` → ACTIVE
  - `cancelledAt != nil` → CANCELLED
- **没有 PAUSED 状态**。`pausedUntil` 字段在首次启动时迁移到 `cancelledAt`(详见迁移章节)

## Requirement 1 —— 模型:加 `cancelledAt`、移除 `pausedUntil` 语义、简化 isActive

### 用户故事

作为工程团队,我们需要一个**单一的活跃 / 已取消真相源**,让每个消费方(首页、详情、Insights、通知、小组件)从同一个字段读。

### 产品行为

- 给 `Subscription` 加 `cancelledAt: Date?`。可空,`nil` 表示活跃
- 现在的 `isActive: Bool` 暂时保留(避免破坏小组件 / 通知里读它的代码),但**改写为基于 `cancelledAt == nil` 的 computed property**。schema 层纯加法,无破坏性 SwiftData 迁移
- `pausedUntil: Date?` 在首次 v1.3 启动时迁移到 `cancelledAt`(详见迁移),然后变成死代码。v1.3 移除它所有的读取点;**保留 stored property** 在模型上,SwiftData 才能继续加载老数据。在 v1.4 或 v1.5(确认该字段已闲置一个版本之后)再正式从模型里删

### 验收标准

- `Subscription.cancelledAt` 通过 SwiftData 持久化往返
- 新增 `Subscription.isCancelled` computed property,返回 `cancelledAt != nil`
- v1.2 里所有 `pausedUntil` 的调用点(通过 grep 验证)被移除或改读 `cancelledAt`
- 订阅取消后再删除,`priceHistory` 仍正常 cascade 删除

---

## Requirement 2 —— 详情页:切换按钮 + 带确认的破坏性删除

### 用户故事

作为用户,我用完一个订阅时,**点一个清晰的按钮就能保留历史**。点错了,**再点一次就翻回来**。只有真心想丢的时候,才会丢数据。

### 产品行为

- 详情页操作区按上面布局渲染两个按钮(亮绿 / 警告红)。按钮文案随状态切换:
  - 活跃:`标记为已取消`(亮绿)+ `删除订阅`(警告红)
  - 已取消:`标记为已订阅`(亮绿)+ `删除订阅`(警告红)
- `标记为已取消` 设 `cancelledAt = .now`,调 `NotificationCoordinator.refresh()`(取消所有挂着的续费通知),更新 `updatedAt`
- `标记为已订阅` 设 `cancelledAt = nil`;如果 `nextBillingDate` 已经在过去,按原周期推进到下一个有效续费日;通过 coordinator 重新调度续费通知
- `删除订阅` 现在**强制一步确认对话框**:`「确定要删除这条订阅?它的历史和价格变动会一起丢失,无法恢复。」`确认后走现有删除路径
- 详情页阅读视图的标题区,在订阅已取消时,**把倒计时替换为 `已取消于 DD MMM YYYY`**,色用 `TrackrColors.fg3`
- 详情页编辑表单对已取消订阅仍可用(用户可能想纠正历史里的名字或金额)

### 验收标准

- 取消 → 恢复 → 取消快速来回切,不留孤儿通知
- 删除订阅仍需确认,点对话框外面取消删除
- 一条已取消、`nextBillingDate` 已过期的订阅,被恢复后,`nextBillingDate` 推进到下一个有效周期日期(不留在「已逾期」)
- 快照测试覆盖两个状态的阅读视图

## Requirement 3 —— 首页:CANCELLED 分组

### 用户故事

作为用户,我在首页底部看到我的已取消订阅,**可以浏览「校友列表」,但它们不会挤占当前视图**。

### 产品行为

- 在 `ACTIVE` 下面新增分组,表头:`CANCELLED` + 计数 + 历来累计花费(显示币种)
- 默认状态:**折叠**。点表头展开
- 计数为 0 时整体隐藏
- 展开后的行用降低透明度渲染(名字用 `TrackrColors.fg2`,金额用 `TrackrColors.fg3`),显示 `「已取消 <DD MMM>」` 替换原本的续费倒计时
- 点一行打开详情(详情页渲染自身的「已取消」版本)

### 验收标准

- 取消第一条订阅后,该 section 立刻出现;最后一条已取消订阅被恢复或删除后,section 立刻消失
- 展开 / 折叠状态是**页面级局部**(下次打开 app 重置)—— 每个 session 默认折叠是有意的
- 点击已取消行无延迟跳详情
- 快照基线覆盖(填充态 + 空 cancelled 组态)

## Requirement 4 —— Insights:过滤已取消订阅 + LIFETIME 栏

### 用户故事

作为 Pro 用户,我的 Insights 数字反映**我现在在付什么** —— 已取消订阅不会虚假抬高当前花费 —— 但我也能看到我**这辈子**在订阅上付过多少,作为我决心走开的那笔财务重量的度量。

### 产品行为

- v1.2 现有的每个 Insights 计算器,**必须跳过** `isCancelled == true` 的订阅:
  - `MonthlyTotalCalculator`(`isActive` 改为 derived 后自动正确,验证一下)
  - `UpcomingChargesCalculator`
  - `SuspectRanker`
  - `CategoryBreakdown`
  - `CurrencyBreakdown`
- 新增纯函数计算器 `LifetimeSpendCalculator.total(of:in:rateTable:)`:**对所有订阅(活跃 + 已取消)**求和,每条贡献 `monthlyEquivalent × monthsBetween(startDate, cancelledAt ?? now)`。返回显示币种下的 `Decimal`
- stat 条加第四栏:`LIFETIME` / `历来累计`,渲染计算器输出
- 这个 stat 读活跃 + 已取消订阅;**已删除的订阅仍然消失**(不会从虚空里复活)

### 验收标准

- 取消一条订阅后,一次渲染周期内从 MONTHLY / YEARLY / NEXT 30 DAYS / suspect / 分类 / 币种里消失
- 同一次取消使 `LIFETIME` 增加 `monthlyEquivalent × (从 startDate 到 cancelledAt 的月数)`
- 恢复订阅反向操作
- `LifetimeSpendCalculator` 有单元测试覆盖:仅活跃、仅已取消、混合、外币换算、缺汇率跳过、空输入

## Requirement 5 —— 迁移:现有 PAUSED 状态 → CANCELLED

### 用户故事

作为已有 TestFlight / App Store 用户、数据里有暂停订阅的人,我的暂停订阅在**首次 v1.3 启动时变成「已取消」** —— 保留它们的数据,在新的 CANCELLED 分组里出现,而不是消失或停在一个已废弃的状态里。

### 产品行为

一次性 backfill 在 app 启动时跑(参考 v1.1 已上线的 `.initial` 价格历史 backfill 模式)。对每条订阅:

| 当前状态 | 迁移后 |
|---|---|
| `isActive == true && pausedUntil == nil` | `cancelledAt = nil`(活跃) |
| `isActive == true && pausedUntil > now` | `cancelledAt = pausedUntil`(已取消,日期锚在用户原本想 pause 结束的那天) |
| `isActive == true && pausedUntil < now` | `cancelledAt = nil`(活跃;pause 已经过期) |
| `isActive == false` | `cancelledAt = updatedAt`(已取消,锚在最近一次编辑) |

迁移**只跑一次,幂等**(用 `UserSettings` 上的 flag 防重,例如 `v1_3_pausedBackfillDoneAt: Date?`)

### 验收标准

- 一个带每种场景的 pre-v1.3 store 正确迁移
- 迁移跑两次结果跟跑一次一样
- 迁移耗时有界(同步、阻塞启动路径,历史上几百条订阅 <100ms)
- 测试为每种 pre-migration 状态构造 SwiftData container,断言迁移后字段值

## 待解问题(开工前必须定)

1. **取消按钮文案。** 拟定:`标记为已取消` ↔ `标记为已订阅`(对齐 Orbit)。备选:`取消订阅` ↔ `重新订阅`、`关闭` ↔ `重启`。推荐拟定版 —— `标记为` 强调是个标签操作,降低用户感知的不可逆性
2. **删除确认文案。** 拟定:`「确定要删除这条订阅?它的历史和价格变动会一起丢失,无法恢复。」`确认或重写
3. **CANCELLED 分组默认状态 —— 折叠还是展开?** 推荐**折叠** —— 保持首页干净;想浏览校友的用户多点一下而已
4. **Insights LIFETIME 栏 —— Pro-gate 还是免费?** 推荐 **Pro-gate**,跟 Insights 其它部分一致 —— Pro 升级故事更连贯(「Pro = 看你的模式,不只是你的现在」)。备选:免费,作为引流
5. **取消时通知怎么办?** 推荐:取消这条订阅时,取消它所有挂着的续费通知。恢复时重新创建。任何边角情况(比如今天早上通知已经发出去了)放过去就好
6. **`isActive == false` 订阅的迁移锚定。** 拟定:`cancelledAt = updatedAt`。备选:`cancelledAt = .now`(锚在迁移那一刻而不是原始停用时间)。推荐 `updatedAt`,历史更真实
