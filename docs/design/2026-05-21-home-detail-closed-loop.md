# 中文版

## 问题陈述

PennyLoop 的首页和详情页都太薄。首页是一个平铺列表加月度总额 hero;详情页是一张字段表(套餐 / 周期 / 分类 / 起始 / 下次 / 状态 / 备注)加暂停 / 删除两个按钮。对比竞品 Orbit,两个页面都做得不够——Orbit 详情页有彩色价格历史记录、清晰的续费状态、取消操作;它的首页把免费试用单独分组,并显示相对续费时间。

## 重新框定 —— 用户到底来干什么

"产品完成度"是开发者视角的感觉,不是用户的 job。真正的 job —— 也是订阅疲劳趋势放大的那个 —— 是:

> 「**这笔周期性扣款,我还要不要继续付?**」

这个决策是**逐笔订阅**做的,而且**发生在详情页**。首页的 job 是**分诊(triage)**:把本月需要注意的东西摆出来,让用户知道该点开哪个详情页。

产品是一个两步闭环:

- **首页回答**:这个月我该看什么?
- **详情页回答**:这一笔我该怎么处理?

现在这个闭环两头都断了——首页不分诊(只是罗列),详情页不支持决策(只是显示字段)。

## 前提(已确认)

1. 目标是 **keep/kill 决策**,不是页面丰富度。两个页面上每加一个元素,都必须对得上这个决策、或对得上导向它的分诊。对不上的装饰一律砍。
2. 首页和详情页作为一个闭环,在 v1.1 一起出。(用户在 office-hours 中选择此项而非"详情页优先"分两步:CC 让第二个页面的边际成本很低,且下次提审前要一次看得见的完成度跃升。)
3. 抄 Orbit 的**信息架构**(价格历史、已取消态、免费试用分组、相对续费时间),**绝不抄它的视觉风格**。PennyLoop 保持像素风:黑 + 亮绿 + VT323。不要行星动画,不要玻璃渐变。
4. 日历页不在本次范围内。它是**新界面**,而首页/详情页是**现有界面**,现有界面太薄是更急的伤口。日历留在 ROADMAP Phase 2。

## 考虑过的方案

- **方案 A 首页优先**:首页做成月度控制台,详情页不动。完成度 5/10。否决——用户注意到了却点进去仍是字段表,闭环在决策端依然断。
- **方案 B 详情页优先**:详情页做成决策页,首页不动。完成度 7/10。office-hours 的推荐项——最低成本交付真正的决策 job,先验证决策页设计再返工首页。未采纳。
- **方案 C 双页闭环一次做完(已采纳)**:首页做分诊,详情页做决策,v1.1 一起出。完成度 9/10。范围更大,但首发即闭环完整。

## 采纳方案 —— C,具体规格

以下全部用现有 PennyLoop 像素风系统(`TrackrColors`、`PixelText`、VT323)渲染,不引入新视觉语言。

### 首页 —— "本月"分诊控制台

保留:`● PENNYLOOP` 字标、FAB、现有的设置/洞察入口。

改 hero。现在是 `MONTHLY · USD` 加一个总额。改成回答"什么时候扣我多少钱":

- **本月剩余扣款** —— 从今天到月底所有续费的总和。这才是引发焦虑的数字。
- **下一笔扣款** —— 最近一笔续费的名称 + 日期 + 金额。

列表分组,而不是一条平铺排序:

- **免费试用 (FREE TRIALS)** 分组,置顶。试用是取消焦虑最高的一类("忘了在转正前取消"),值得单独成区。
- **进行中 (ACTIVE)** 分组在下面。
- 以后可选:按计费周期分组(月付 / 年付)—— 除非很 trivial,否则推迟。

每一行加一条**相对续费**文案:"RENEWS IN 3 DAYS",和现有日期并排。Orbit 这么做,比光给日期读得快。

首页**不要**的东西:为丰富而丰富的 stat 卡片网格、图表、行星元素。分诊是 job,四张装饰卡不是。

### 详情页 —— "单笔订阅决策"页

围绕 keep/kill 决策重构 `SubscriptionDetailView`:

- **页头** —— 品牌图标 + 名称 + 金额,比现在做大。这是你正在决策的对象的身份。
- **续费倒计时** —— "RENEWS IN 3 DAYS" 作为醒目的一行,不要埋在字段表里。时间压力跟决策相关。
- **字段** —— 周期、分类、起始、下次、状态。保留,但次于倒计时和价格历史。
- **价格历史** —— 价值最高的新增项。PennyLoop 已经在跑价格变动检测,数据已有。渲染一个逐订阅的历史列表,带配色涨跌:跌用亮绿、涨用警示红,每行 `日期 · ↑/↓ 变动 · 新价格`。这直接服务"catch price changes"卖点,回答"它是不是越来越贵了"—— 这是 keep/kill 的核心输入。几乎零新基建:把已有数据呈现出来。
- **备注** —— 保留。
- **URL** —— 改成可点击链接。点击打开该服务的网站 / 管理页,让用户能就决策行动(去取消、去降级)。
- **操作** —— "MARK AS CANCELLED" 和 "DELETE" 区分开。已取消 = 订阅结束但保留用于历史和洞察;删除 = 彻底没了。与现有的 PAUSE 操作如何调和,见下面的待解问题。

### 闭环,具体走一遍

首页:"Netflix · RENEWS IN 3 DAYS · ¥58" 在 ACTIVE 分组里 → 点击 → 详情页:倒计时 + 价格历史显示今年涨了两次 + 可点的 netflix.com + MARK AS CANCELLED。用户看到扣款将至、看到它一直在涨、然后行动。分诊 → 决策。

## 待解问题(开工前必须定)

1. **PAUSE 与 MARK AS CANCELLED。** PennyLoop 的 `Subscription` 模型和详情页现在有 PAUSE。已取消和已暂停是不同状态(暂停 = 临时不活跃、仍是你的;已取消 = 结束了)。三选一:(a) 两个状态都保留为独立态,(b) 用 CANCELLED 替换 PAUSE,(c) 保留 PAUSE 再加 CANCELLED。这会影响 `Subscription` 模型、状态枚举、快照测试,以及每个状态如何计入月度总额。工程开工前定。
2. **价格历史的存储。** 确认逐订阅的价格变动是否已经持久化(SwiftData),还是只在检测过程中临时算出来的。如果是临时的,详情页的价格历史列表需要一个小的持久化模型。这是"几乎零新基建"唯一可能不成立的地方,**先核实**。
3. **"本月剩余"的多币种处理。** hero 的求和必须像现有月度总额那样处理多币种订阅(FX 锁定汇率、归一到主货币)。复用 `MonthlyTotalCalculator` 的货币逻辑,不要重造。

## 成功标准

- 打开首页,不用点击就能回答"这个月什么时候扣我多少钱"。
- 免费试用在首页与进行中的订阅视觉上分开。
- 详情页显示续费倒计时;任何有价格变动记录的订阅,显示配色涨跌的价格历史列表。
- 用户能从"在首页注意到"一路走到"做出决策并行动"(取消 / 打开管理页),不脱离闭环。
- 全部用现有像素风系统渲染,不引入新视觉语言。
- 首页和详情页新布局的快照测试 baseline 更新。

# English Version

## Problem Statement

PennyLoop's Home and Detail screens are thin. Home is a flat list with a
monthly-total hero; Detail is a field table (PLAN / CYCLE / CATEGORY /
STARTED / NEXT / STATUS / NOTES) plus PAUSE / DELETE. Compared to the
competitor Orbit, both screens under-deliver — Orbit's detail page carries
a color-coded price-history log, a clear renewal state, and a cancel
action; its home groups free trials separately and shows relative renewal
timing.

## The Reframe — what the user is actually here to do

"Product completeness" is a builder-side feeling, not a user job. The real
job, and the one the subscription-fatigue trend sharpens, is:

> "This recurring charge — do I keep paying for it or not?"

That decision is per-subscription and it happens on the **Detail** screen.
Home's job is **triage**: surface what needs attention this month so the
user knows which Detail screen to open.

The product is a two-step loop:

- **Home answers:** what should I look at this month?
- **Detail answers:** what do I do about this one?

Today the loop is broken at both ends — Home doesn't triage (it just lists),
and Detail doesn't support a decision (it just displays fields).

## Premises (agreed)

1. The goal is the **keep/kill decision**, not screen richness. Every new
   element on either screen must map to that job or to triage toward it.
   Decoration that doesn't is cut.
2. Home and Detail ship together as one closed loop in v1.1. The user
   chose this over a detail-first split: with Claude Code the marginal
   cost of the second screen is low, and a single visible completeness
   jump is worth more before the next App Store push.
3. Borrow Orbit's **information architecture** (price history, cancelled
   state, free-trials grouping, relative renewal timing), never its
   visual style. PennyLoop stays pixel-art: black + lime + VT323. No
   planet animation, no glossy gradients.
4. The Calendar screen is out of scope here. It is a *new* surface;
   Home/Detail are *existing* surfaces and their thinness is the more
   acute wound. Calendar stays in ROADMAP Phase 2.

## Approaches Considered

- **Approach A: Home-first** — make Home a monthly console, leave Detail
  as-is. Completeness 5/10. Rejected — the user notices something needs
  attention but tapping in still dead-ends in a field table.
- **Approach B: Detail-first** — make Detail a decision page, leave Home
  as-is. Completeness 7/10. The office-hours recommendation; not chosen.
- **Approach C: Both pages, closed loop (CHOSEN)** — Home becomes triage,
  Detail becomes decision, shipped together in v1.1. Completeness 9/10.

## Recommended Approach — C, specified

Everything below renders in the existing PennyLoop pixel-art system
(`TrackrColors`, `PixelText`, VT323). No new visual language.

### Home — "this month" triage console

Keep: `● PENNYLOOP` wordmark, the FAB, existing nav to Settings/Insights.

Change the hero. Today it is `MONTHLY · USD` + a single total. Make it
answer "what hits my card, and when":

- **This month's remaining charges** — sum of renewals from today through
  end of month. This is the number that drives anxiety.
- **Next charge** — name + date + amount of the nearest upcoming renewal.

Group the list instead of one flat sort:

- **FREE TRIALS** group, surfaced first. Trials are the highest
  cancel-anxiety item ("forgot to cancel before it converted"). They earn
  their own section.
- **ACTIVE** group below.
- Optional later: group-by billing cycle (monthly / yearly) — defer unless
  trivial.

Each row gains a **relative renewal line**: "RENEWS IN 3 DAYS" alongside
the existing date. Orbit does this; it reads faster than a bare date.

What Home does NOT get: stat-card grids for their own sake, charts, the
planet motif. Triage is the job; four decorative cards is not.

### Detail — "single-subscription decision" page

Restructure `SubscriptionDetailView` around the keep/kill decision:

- **Header** — brand icon + name + amount, set larger than today.
- **Renewal countdown** — "RENEWS IN 3 DAYS" as a prominent line, not
  buried in the field table. Time pressure is decision-relevant.
- **Fields** — cycle, category, started, next, status. Keep, but
  secondary to the countdown and price history.
- **Price history** — the highest-value add. PennyLoop already runs
  price-change detection; the data exists. Render a per-subscription
  history list with color-coded deltas: lime for a decrease, warn/red for
  an increase, each row `date · ↑/↓ delta · new price`. Directly serves
  the "catch price changes" value prop and answers "is this getting more
  expensive?" — a core input to keep/kill. Near-zero new infrastructure.
- **Notes** — keep.
- **URL** — make it a tappable link to the service's site / manage page.
- **Actions** — "MARK AS CANCELLED" distinct from "DELETE". Cancelled =
  ended but kept for history and insights; Delete = gone entirely. See
  Open Questions on reconciling with the existing PAUSE action.

### The loop, concretely

Home: "Netflix · RENEWS IN 3 DAYS · ¥58" in the ACTIVE group → tap →
Detail: countdown + price history showing two increases this year +
tappable netflix.com + MARK AS CANCELLED. Triage → decide.

## Open Questions (must resolve before eng work)

1. **PAUSE vs MARK AS CANCELLED.** `Subscription` and the detail page
   currently have PAUSE. Cancelled and Paused are different states.
   Decide: (a) keep both as distinct states, (b) replace PAUSE with
   CANCELLED, (c) keep PAUSE, add CANCELLED. Affects the `Subscription`
   model, status enums, snapshot tests, and how each state rolls into the
   monthly total.
2. **Price-history storage.** Confirm whether per-subscription price
   changes are already persisted (SwiftData) or only computed transiently
   by the detection pass. If transient, the Detail price-history list
   needs a small persisted model. Verify first — this is the one place
   "near-zero infra" may not hold.
3. **"Remaining this month" across currencies.** The hero sum must handle
   multi-currency subscriptions like the existing monthly total
   (FX-pinned, home currency). Reuse `MonthlyTotalCalculator`.

## Success Criteria

- Opening Home answers "what hits my card this month and when" without a
  tap.
- Free trials are visually separated from active subscriptions on Home.
- Detail shows a renewal countdown and, for any subscription with a
  recorded price change, a color-coded price-history list.
- A user can go from "noticed on Home" to "decided and acted" without
  leaving the loop.
- All work renders in the existing pixel-art system.
- Snapshot baselines updated for the new Home and Detail layouts.

## Distribution Plan

submit. No new distribution channel. Ships only after 1.0.1 is approved
and live.

## Dependencies
