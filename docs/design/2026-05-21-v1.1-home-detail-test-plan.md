# PennyLoop v1.1 首页 + 详情页闭环测试方案

> 依据: `docs/design/2026-05-21-home-detail-closed-loop.md`  
> 目标版本: v1.1  
> 测试目标: 验证 PennyLoop 从“能记录订阅”升级为“能帮助用户判断一笔订阅是否继续付费”的核心闭环。

---

## 1. 测试总目标

v1.1 的核心不是“页面变丰富”，而是验证一个闭环是否成立:

1. 首页能完成分诊: 用户不用点开详情，就知道本月哪些订阅值得注意。
2. 详情页能支持决策: 用户点进单个订阅后，能判断这笔钱是否继续付。
3. FX 重做后，首页金额不能少算或跳变。
4. 价格历史新增后，创建、编辑、迁移路径都不能丢数据。
5. 现有 1.0/1.0.1 能力不能回退: 添加订阅、编辑、暂停、删除、提醒、付费入口、反馈入口都要继续可用。

---

## 2. 测试分层策略

### 2.1 单元测试

覆盖纯业务逻辑，要求不依赖 SwiftUI 渲染、不依赖真实网络、不依赖系统当前日期。

重点对象:

- 年/月花费计算
- FX 汇率表和跨币种换算
- 年度金额锚物匹配
- 相对续费文案
- 首页分组规则
- 价格历史写入规则
- 价格历史涨跌 delta 计算

### 2.2 SwiftData / Repository 测试

覆盖新增模型、迁移兼容、写入路径。

重点对象:

- `PriceHistoryEntry`
- FX rate cache 模型或缓存 JSON
- 老用户已有 `Subscription` 的兼容读取
- v1.1 首次启动的历史基线补写

### 2.3 View Snapshot 测试

覆盖首页和详情页新布局是否稳定。

只用 snapshot 验证视觉结构，不用它替代业务逻辑测试。

重点对象:

- 首页空状态
- 首页有免费试用 + active 分组
- 首页多币种年度 hero
- 详情页 active 状态
- 详情页 paused 状态
- 详情页有价格历史
- 详情页只有初始价格，无价格变动
- 详情页有未读 `PriceChangeAlert` banner

### 2.4 端到端手工测试

覆盖真实用户路径、App Store 提审前 smoke、设备差异。

重点路径:

- 首页发现即将续费 → 进入详情 → 查看价格历史 → 打开服务 URL / 暂停 / 删除
- 设置切换显示币种 → 回首页金额正确重算
- 编辑订阅金额 → 详情页价格历史出现新记录

## 4. 手工验收场景

### M-001: 真实 keep/kill 路径

1. 创建 Netflix，金额 $15.49，3 天后续费，URL 填 netflix.com。
2. 打开首页。
3. 确认 Netflix 在 `ACTIVE` 分组靠前，显示 `RENEWS IN 3 DAYS`。
4. 点进详情。
5. 确认详情页能看到倒计时、金额、历史、URL、PAUSE/DELETE。
6. 点击 URL，确认能跳到外部管理页。
7. 回到 app，暂停该订阅。
8. 回首页确认状态变化。

通过标准: 用户能从首页注意到问题，再在详情页完成判断和行动。

### M-002: 免费试用焦虑路径

1. 创建一个 4 天后试用结束的 Spotify。
2. 首页应优先显示 `FREE TRIALS`。
3. 点进详情，确认倒计时比普通字段更醒目。
4. 暂停或删除后，首页分组更新。

通过标准: 免费试用不会淹没在 active 列表里。

### M-003: 多币种切换路径

1. 创建 USD、CNY、EUR 三种币种订阅。
2. 首页显示默认 USD 年度花费。
3. 到 Settings 切成 CNY。
4. 回首页。
5. 确认三种订阅都被计入总额。
6. 断网后重启 app，确认 fallback/缓存仍能显示合理金额。

通过标准: 切换显示币种后不丢订阅、不显示 0、不阻塞 UI。

### M-004: 价格历史路径

1. 创建 Notion $8。
2. 进入详情，确认无价格变动或只有初始基线。
3. 编辑金额为 $10。
4. 回详情，确认出现涨价历史。
5. 再编辑备注，不改金额。
6. 确认历史没有新增一条无意义记录。

通过标准: 价格历史只记录真正价格/币种变化。

### M-005: 旧数据迁移路径

1. 安装上一版本，创建 3 条订阅。
2. 升级到 v1.1。
3. 打开首页，确认数据仍在。
4. 进入详情，确认旧订阅有 initial 基线或能在下次改金额后开始累积历史。
5. 重启 app，确认不会重复补写 initial。

通过标准: 老用户升级无数据丢失，无重复历史。

## 6. 发布门禁

v1.1 不应进入 TestFlight / App Store，除非满足以下条件:

- 首页和详情页 snapshot baseline 已更新且稳定。
- FX 切换显示币种的旧 bug 有明确单测覆盖。
- 新增 `PriceHistoryEntry` 的创建、编辑、迁移、删除策略都有测试。
- 首页分组和相对续费文案不依赖真实当前日期，测试里固定 clock/calendar。
- 手工验收覆盖至少一台真实 iPhone。
- 旧版本升级到 v1.1 后数据不丢失。
- App Store 截图不出现内部 debug 文案或已下线功能。

## 7. 给实现阶段的测试性要求

为了让 Claude Code 实现时更容易写对测试，建议在 PRD / implementation plan 里明确这些边界:

1. 首页分组不要直接写在 `HomeView.body` 里，抽成 `HomeSectionBuilder` 或等价 helper。
2. 相对续费文案抽成纯函数，传入 `now` 和 `calendar`。
3. 年/月花费计算抽成纯函数，传入订阅数组、目标币种、FX rate table。
4. 锚物选择抽成纯函数，传入 annualSpend 和 catalog。
5. 价格历史写入不要散落在 View 里，使用 `PriceHistoryRepository` / service。
6. v1.1 首次补 initial 必须幂等，有独立状态或通过查询现有记录保证。
7. FX 刷新必须是后台行为，UI 读取缓存，不等待网络。

这些要求不是额外架构洁癖，而是为了避免 v1.1 的核心逻辑只能靠截图和手工测试兜底。
