# PennyLoop v1.0.1 Requirements — Feedback Entry

Status: revised after Claude Code review. This is the first post-review update scope. Do not modify the App Store build that is currently in review unless Apple rejects it or a critical launch blocker appears.

## Decision

v1.0.1 should ship one user-visible improvement:

1. A high-visibility feedback/support entry inside Settings.

Analytics scaffolding is explicitly deferred to v1.0.3 or later. A release-only no-op analytics layer creates implementation and review surface without real user value, so it should not be part of v1.0.1.

## Release Version Rule

Before archiving v1.0.1:

- Set `MARKETING_VERSION` to `1.0.1`.
- Set `CURRENT_PROJECT_VERSION` to a build number greater than the latest uploaded App Store Connect build.
- After the first v1.0.1 review attempt, `project.yml` is being moved to build `15` for resubmission. Future uploads must use a higher build number than the latest uploaded App Store Connect build.

Do not reuse build numbers. App Store Connect rejects reused build numbers.

## Non-Goals for v1.0.1

- Do not add Firebase, PostHog, Amplitude, Segment, AppsFlyer, Adjust, or any other third-party analytics SDK.
- Do not add an analytics abstraction in v1.0.1.
- Do not add login, account creation, user profile, or server-side identity.
- Do not upload subscription names, prices, notes, URLs, OCR text, screenshots, or category details.
- Do not change the current one-time Lifetime IAP model.
- Do not change App Store metadata for the build currently in review.

## Requirement — Feedback Entry

### User Story

As a user, I can find feedback/support from the part of Settings I actually look at, and I can choose a channel that works for me without needing a GitHub account.

### Placement

Add a Feedback section near `PRO STATUS`, not buried below Privacy Policy and Terms of Service.

Recommended order inside `SettingsView`:

1. Import Existing Subs
2. Pro Status
3. Feedback
4. Privacy Policy / Terms of Service

The legal links remain at the bottom. Feedback should be visually near the Pro/payment area because restore purchase, billing, and screenshot import issues are the most likely early support requests.

### UI

Add a section label:

- English: `FEEDBACK`
- Simplified Chinese: `意见反馈`

Under it, expose three actions:

1. Email Feedback
2. Rate on App Store
3. GitHub Issue

Recommended labels:

| Action | English | Simplified Chinese |
|---|---|---|
| Email | `EMAIL FEEDBACK` | `邮件反馈` |
| App Store rating | `RATE ON APP STORE` | `App Store 评分` |
| GitHub issue | `GITHUB ISSUE` | `GitHub Issue` |

Keep the existing PennyLoop visual system: pixel label, simple rows/buttons, lime accent for actionable text, no new card surface.

### Action Behavior

#### Email Feedback

Use `mailto:` via `openURL`, not a third-party SDK.

The email address must be configured before implementation lands. Add this constant:

```swift
static let supportEmail = "TODO_REPLACE_WITH_REAL_SUPPORT_EMAIL"
```

Implementation must not ship with the `TODO_` placeholder. Add a test that fails while the placeholder remains.

Subject format:

```text
PennyLoop Feedback - <version> (<build>)
```

Body should prefill only non-sensitive diagnostic context:

```text
Please describe what happened:


---
App: PennyLoop
Version: <CFBundleShortVersionString>
Build: <CFBundleVersion>
iOS: <UIDevice.current.systemVersion>
Language: <resolved app language or Locale.current.identifier>
Pro Status: <free/proLifetime>
```

Do not include:

- Subscription names
- Amounts
- Renewal dates
- Notes
- URLs
- OCR text
- Screenshot contents
- Device model
- Storefront
- Timezone

#### Rate on App Store

Use Apple's in-app review flow if possible:

- Prefer `SKStoreReviewController.requestReview(in:)` when a foreground `UIWindowScene` is available.
- If implementation needs a URL fallback, add an App Store product URL only after the App Store app ID is known.

Do not block the Feedback section if rating cannot be opened on simulator/TestFlight.

#### GitHub Issue

Open:

```text
https://github.com/Drinkwater0922/subscription-management-/issues/new
```

This is a fallback for technical users, not the primary path.

### BrandConfig Changes

Add:

```swift
static let supportURL = URL(string: "https://drinkwater0922.github.io/subscription-management-/support/")!
static let supportIssueURL = URL(string: "https://github.com/Drinkwater0922/subscription-management-/issues/new")!
static let supportEmail = "TODO_REPLACE_WITH_REAL_SUPPORT_EMAIL"
```

Optional later, once the App Store ID is known:

```swift
static let appStoreReviewURL = URL(string: "itms-apps://itunes.apple.com/app/id<APP_ID>?action=write-review")!
```

For v1.0.1, `SKStoreReviewController` is preferred over hard-coding an App Store review URL.

### Localization

Update `Trackr/Resources/Localizable.xcstrings` with:

- `FEEDBACK`
- `EMAIL FEEDBACK`
- `RATE ON APP STORE`
- `GITHUB ISSUE`

Simplified Chinese values:

- `意见反馈`
- `邮件反馈`
- `App Store 评分`
- `GitHub Issue`

### Acceptance Criteria

- Settings shows a Feedback section near Pro Status.
- Email Feedback opens a mail URL with version/build/iOS/language/pro-status context.
- Email body contains no subscription content and no screenshot/OCR content.
- Rate on App Store uses Apple's review controller or a safe fallback once App ID is known.
- GitHub Issue opens the issue creation URL.
- Privacy Policy and Terms of Service remain available.
- Existing Settings behavior for restore purchases, Pro status, privacy, and terms still works.
- v1.0.1 archive uses `MARKETING_VERSION = 1.0.1` and a new build number greater than the current App Store Connect build.

## Deferred Analytics Scope — v1.0.3 or Later

Analytics should return only when we are ready to collect real product signal and update privacy disclosures if needed.

### Analytics Decision Gate

Do not implement analytics until at least one of these is true:

- App Store reviews / support requests show a recurring funnel problem that cannot be understood from qualitative feedback.
- Search Ads or ASO tests create enough traffic that conversion measurement is useful.
- A remote collection path is selected and privacy disclosures are updated before submission.

### Future Event Candidates

If analytics is added later, start from this event list:

| Event | Why it matters |
|---|---|
| `app_launched` | Basic active use |
| `home_opened` | Retention signal; limit to once per day |
| `onboarding_completed` | Activation |
| `subscription_added` | Core value reached |
| `subscription_deleted` | Possible churn or cleanup signal |
| `photo_import_started` | Import intent |
| `photo_import_succeeded` | Import value |
| `photo_import_failed` | Import friction |
| `notification_permission_denied` | Core reminder value blocked |
| `renewal_notification_tapped` | Reminder effectiveness |
| `paywall_shown` | Monetization exposure |
| `purchase_started` | Purchase intent |
| `purchase_succeeded` | Monetization success |
| `purchase_failed` | Monetization friction |
| `restore_started` | Purchase recovery friction |
| `restore_succeeded` | Purchase recovery success |
| `widget_added` | Strong retention signal, only if reliably detectable |
| `feedback_tapped` | Support intent |

### Future Allowed Properties

Only allow constrained enum/string values:

- `app_version`
- `build`
- `locale`
- `pro_status` (`free`, `proLifetime`)
- `source` (`manual`, `preset`, `photo`, `settings`, `subscription_limit`, `insights_locked`, `manual_upgrade`)
- `result` (`success`, `failed`, `cancelled`, `empty`)
- `error_code` as a closed enum, not arbitrary strings

### Future Forbidden Properties

Never include:

- Subscription name
- Subscription price or amount
- Currency for a specific subscription
- Billing date or renewal date
- Category
- Notes
- URL
- OCR text
- Screenshot image data
- User email
- Apple ID
- Device advertising ID
- Persistent custom user ID
- IP address collection by app code
- Timezone
- Device model
- Storefront
- Arbitrary error message text

`error_code` must be a closed enum so sensitive OCR or StoreKit text cannot leak into analytics as a raw string.

### Future Backend Options

When analytics becomes justified, choose one deliberately:

- A minimal first-party endpoint with NDJSON ingestion.
- A privacy-focused analytics service after updating privacy disclosures.
- A no-remote, local-only debug trace if the goal is QA rather than production analytics.

Do not use CloudKit private database for developer-wide analytics aggregation. Private CloudKit data belongs to each user's iCloud account and is not an appropriate collection path for cross-user funnel analytics.

## Review Checklist

The v1.0.1 implementation is acceptable when:

- Feedback is visible without scrolling to the legal footer.
- Email is the primary feedback channel and does not require GitHub.
- GitHub Issue remains available as a technical fallback.
- Rating action uses Apple-supported review behavior.
- No analytics code or SDK is introduced.
- No privacy policy or App Privacy change is needed for v1.0.1.
- Build/version numbers are safe for App Store Connect upload.
