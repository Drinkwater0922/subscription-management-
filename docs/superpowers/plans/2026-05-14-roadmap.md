# TRACKR — Implementation Roadmap

> **Working name:** TRACKR (placeholder; final name TBD)
> **Spec:** [/docs/superpowers/specs/2026-05-14-subscription-tracker-design.md](../specs/2026-05-14-subscription-tracker-design.md)

## Goal

Ship an iOS App Store launch (V1) of a subscription tracker positioned for the AI / developer niche. Pure Apple stack, no third-party dependencies, iOS 17 minimum.

## Why milestones, not one mega-plan

This product spans ~9 distinct subsystems (data layer, UI, notifications, preset sync, IAP, widgets, CloudKit, onboarding, launch ops). Each is large enough to be its own plan. They are sequential — later milestones depend on earlier ones — but each milestone produces a working, testable build of the app that we can run, test, and learn from before writing the next plan.

The next milestone's plan is written only after the previous one ships. This avoids writing detailed task-by-task plans for code we don't yet understand the constraints of.

## Milestones

### M1 — Foundation _(plan written, ready to execute)_
**Detailed plan:** [2026-05-14-milestone-1-foundation.md](./2026-05-14-milestone-1-foundation.md)
**Deliverable:** Buildable Xcode project. Design system (colors / typography / 4 base components) implemented with unit tests. Empty `Home` screen renders the design system pieces. Runs in simulator on iOS 17.
**Acceptance:** `Cmd+R` launches the app; a styled placeholder Home screen with hero number, dashed divider, and FAB is visible; `Cmd+U` runs the test suite green.

### M2 — Data layer
**Deliverable:** SwiftData models (`Subscription`, `RenewalEvent`, `PriceChangeAlert`, `UserSettings`, `PresetCache`) with repository wrappers. Currency / amount formatting utilities. Cycle math (next renewal date computation, anchored on `startDate` to prevent drift). All logic TDD'd against in-memory `ModelContainer`.
**Acceptance:** Repositories pass full unit-test coverage; cycle math handles Feb-29 anchor, 31-of-month anchor, leap years correctly.

### M3 — Core CRUD UI
**Deliverable:** Home screen with real subscriptions list bound to SwiftData (`@Query`). Add Subscription sheet (CUSTOM tab only — preset library lands in M5). Subscription Detail screen with edit / pause / cancel / delete. Empty state. Hero monthly total computed from data.
**Acceptance:** Can create, view, edit, and delete a subscription in the simulator end-to-end. Hero total reflects edits live.

### M4 — Notifications & cycle math integration
**Deliverable:** `LocalNotificationScheduler` wraps `UNUserNotificationCenter`. Auto-schedules per-`leadDay` notifications on subscription create / edit. Same-day aggregation. Deep-link tap routes to detail. Settings screen shows notification preferences.
**Acceptance:** Create a subscription with `nextBillingDate` 2 minutes away → notification fires at the right time → tap → app deep-links to that subscription's detail screen. Edit the date → old notification cancelled and replaced.

### M5 — Preset library + AI price-change alerts
**Deliverable:** Bundled `presets.bundled.json` (seed list, ~60 items curated during this milestone). `PresetFetcher` (URLSession remote fetch), `PriceChangeDiffer` (cache-vs-remote diff producing `PriceChangeAlert` records). LIBRARY tab in Add Subscription sheet with search and category grouping. In-app price-change banner on Detail screen.
**Acceptance:** Mock a `presets.json` version bump locally → trigger fetch → an existing subscription's Detail screen shows the inline alert banner. LIBRARY tab pre-fills the Confirm form correctly.

### M6 — IAP, paywall, free / Pro gating
**Deliverable:** `ProEntitlement` wrapping StoreKit 2. `FeatureGate` static map of feature → required tier. `PaywallTriggerCoordinator`. PaywallView. `.storekit` configuration file for local sandbox testing. Free 5-subscription limit enforced. Pro-only price-change push notification (the M5 alerts get pushed only when Pro). Insights screen scaffolded (V1: totals only, gated behind Pro).
**Acceptance:** Local StoreKit test buys monthly + lifetime products successfully; entitlement state flips; gated views update without restart; paywall is presented when triggered.

### M7 — Widget + iCloud sync
**Deliverable:** Widget extension target with Small + Medium widget types (next-renewal countdown). App Group + shared SwiftData container for widget read access. CloudKit container configured; sync turned on when user is Pro and signed into iCloud. Family Sharing toggles verified.
**Acceptance:** Widget added to home screen renders real data. Two simulator instances signed into the same iCloud test account sync subscriptions within ~10 seconds.

### M8 — Onboarding, localization, polish
**Deliverable:** 3-screen onboarding (brand / value pitch / notification permission). Localization: `zh-Hans` + `en` string catalogs. App icon (pixel-style monogram). Settings screen complete (language toggle, currency, lead days, hour, Pro status, links). Empty-state polish. Sensory feedback on key interactions.
**Acceptance:** Cold launch flows through onboarding correctly; switching system language between zh-Hans and English reflows the UI; app icon renders on home screen at all sizes.

### M9 — Pre-launch: name lock, legal, App Store assets, beta
**Deliverable:** Final app name locked (after trademark / App Store search / domain check done in parallel). Privacy policy + Terms URLs published. App Store Connect record created with screenshots, description, keywords, screenshots in zh-Hans + en. TestFlight beta build pushed to internal testers. Submission ready.
**Acceptance:** Internal testers can install via TestFlight on real devices; all critical bugs from beta resolved; app passes Apple's App Store Review Guideline self-checklist.

## Critical path total

M1 → M2 → M3 → M4 → M5 → M6 → M7 → M8 → M9.

M7 (Widget) and M8 (onboarding/polish) can technically swap order if needed, but the default is sequential.

## When does the next plan get written?

At each milestone's "Acceptance" gate. We do not write M2's plan until M1 ships and we've learned what Xcode actually wants. This keeps plans grounded.
